# Running `agent-langgraph` locally — what works, what doesn't

## TL;DR

**You can run the agent locally with zero code changes. You do NOT need to deploy to Databricks first.** The template is explicitly designed for a "local dev loop → deploy later" workflow.

```bash
cd /Users/javier.delacruz/.cline/worktrees/fafcd/app-business-intelligence/agent-langgraph
uv run quickstart      # one-time setup (auth + MLflow experiment)
uv run start-app       # starts backend on :8000 and chat UI on :3000
```

That's it. The same code path runs locally and on Databricks Apps — `databricks.yml` and `app.yaml` use the **exact same start command** (`uv run start-app`). So if it works locally, the deploy path is essentially identical.

| Requirement | Why |

## What works LOCALLY (out of the box)

✅ **The full FastAPI agent server** — `agent_server/start_server.py` runs as a normal Python/uvicorn process on `localhost:8000`. The `/invocations`, `/responses`, `/health` endpoints all work.

✅ **The LangGraph agent itself** — `create_agent(...)` in `agent_server/agent.py` runs in your local Python process.

✅ **The LLM call** — `ChatDatabricks(endpoint="databricks-gpt-5-2")` works locally because the Databricks SDK auto-authenticates from your `.env` (`DATABRICKS_CONFIG_PROFILE` or `DATABRICKS_HOST` + `DATABRICKS_TOKEN`) and calls the model serving endpoint over HTTPS. Identical behavior to deployed.

✅ **The built-in `get_current_time` tool** — pure local Python, no dependency on anything.

✅ **Streaming responses** — `/invocations` with `"stream": true` works locally.

✅ **MLflow tracing to Databricks** — `MLFLOW_TRACKING_URI=databricks` + your `MLFLOW_EXPERIMENT_ID` sends traces from your laptop to the workspace experiment. You'll see them in the MLflow UI in your workspace.

✅ **The Next.js chat UI** at `http://localhost:3000` (auto-cloned by `start_app.py` from the public `databricks/app-templates` repo).

✅ **`uv run preflight`** — automated end-to-end test (start server → `/health` → `/invocations`).

✅ **`uv run agent-evaluate`** — runs `evaluate_agent.py` locally; the `ConversationSimulator` uses `databricks:/databricks-claude-sonnet-4-5` as the user model, which also works locally as long as your profile has access.

✅ **MCP tools (if you uncomment them)** — `DatabricksMCPServer(url=f"{host}/api/2.0/mcp/functions/system/ai", ...)` works locally too. The MCP server is hosted on the workspace; your local agent just makes HTTPS calls to it. **Including Genie**, **UC Functions**, **Vector Search** — all work locally because they're called over the same workspace API your service principal would use when deployed.

## What's DIFFERENT or LIMITED locally

⚠️ These are behavior differences, not blockers. The code itself doesn't change between local and deployed; the runtime context does.

### 1. Authentication identity
- **Local:** calls Databricks as **you** (your OAuth user / PAT) via `WorkspaceClient()` reading your profile.
- **Deployed:** calls Databricks as the **app's service principal**.
- **Implication:** A tool call that succeeds locally may fail after deploy if the service principal doesn't have the same Unity Catalog / Genie / endpoint permissions as you. The fix is in `databricks.yml` `resources:` (which grants the SP access at deploy time) — but you can't fully test SP-mode permissions until you deploy.

### 2. On-Behalf-Of (OBO) user authentication does NOT work locally
- Look at `agent_server/utils.py:24-26`:
  ```python
  def get_user_workspace_client() -> WorkspaceClient:
      token = get_request_headers().get("x-forwarded-access-token")
      return WorkspaceClient(token=token, auth_type="pat")
  ```
- The `x-forwarded-access-token` header is injected by Databricks Apps' reverse proxy. **It is not present when you call `localhost:8000` directly.** So if you switch on the OBO line in `agent.py` (`get_user_workspace_client()`), local calls will fail or fall back silently.
- ✅ Workaround for local dev: use the default `sp_workspace_client` path (it's the default — keep it commented out).

### 3. The `experiment` resource binding
- `app.yaml` has `MLFLOW_EXPERIMENT_ID: valueFrom: "experiment"` — the value is resolved by Databricks Apps from the bound resource at runtime.
- **Locally** the experiment ID is read from your `.env` file (`MLFLOW_EXPERIMENT_ID=...`). `quickstart` writes it for you. Functionally equivalent, just a different source.

### 4. Resource bindings in `databricks.yml` are not enforced locally
- The `resources:` block in `databricks.yml` (experiment, UC functions, Genie spaces, serving endpoints, Lakebase) grants the deployed app's SP access. Locally, those grants are irrelevant — your user already has whatever access your user has.
- **Implication:** A tool can work locally but fail post-deploy because you forgot to add it to `databricks.yml`. The `add-tools` skill in `.claude/skills/` exists specifically to keep these in sync.

### 5. Network egress / proxies
- Locally, your laptop calls `*.databricks.com` directly.
- Deployed, the app runs inside Databricks' network with potentially different egress rules. Rarely an issue, but if your agent hits external APIs, behavior may differ.

### 6. Lakebase / Postgres-backed memory (only relevant if you add it later)
- This template doesn't include Lakebase. If you later add agent memory backed by Lakebase (per the `agent-memory` skill), local connections need `PGHOST`/`PGUSER`/`PGDATABASE` in `.env` and that auth flow differs from in-app SP auth. The `quickstart` script handles the local side for you.

### 7. Querying the agent itself
- **Locally:** `curl http://localhost:8000/invocations -d '{...}'` — no auth required.
- **Deployed:** must hit `https://<app>.databricksapps.com/responses` with an OAuth token (PATs are not accepted; see README §5).

### 8. Auto-reload / multiple workers
- `uv run start-server --reload` (hot reload) works locally only.
- `uv run start-server --workers 4` works both places, but Databricks Apps controls process lifecycle.

### 9. App startup environment variables
- `app.yaml` and `databricks.yml` define the env vars Databricks Apps sets at runtime (`API_PROXY`, `CHAT_APP_PORT`, etc.).
- Locally, `start_app.py` calls `load_dotenv(".env")` and sets `API_PROXY` programmatically (line 222). Equivalent end result.

## Things you literally cannot test locally

🚫 **App-level configuration / secrets injection** from Databricks Apps (`valueFrom:` in `app.yaml`) — locally you must set the env var yourself in `.env`.

🚫 **The deployed URL / OAuth flow** — only exists after `databricks bundle deploy && databricks bundle run`.

🚫 **Service principal permissions** — see #1 above. The only way to test SP-mode is to deploy.

🚫 **Databricks Apps autoscaling / process supervision** — the platform restarts your app on failure, applies CPU/memory limits, etc. Locally `uvicorn` just runs.

🚫 **The "MLflow experiment binding"** — the `value_from: "experiment"` mechanism only resolves on Databricks. Locally you wire the ID via `.env` instead.

## Recommended local workflow (matches the README)

```bash
# 1. One-time setup — creates .env, MLflow experiment, sets profile
uv run quickstart

# 2. Run the app (backend + chat UI)
uv run start-app
# → http://localhost:8000  (API: POST /invocations)
# → http://localhost:3000  (chat UI)

# 3. Backend only (faster iteration, no Next.js)
uv run start-app --no-ui
# or with hot reload:
uv run start-server --reload

# 4. End-to-end smoke test before deploy
uv run preflight

# 5. Smoke-test from curl
curl -X POST http://localhost:8000/invocations \
  -H "Content-Type: application/json" \
  -d '{"input":[{"role":"user","content":"What time is it?"}]}'

# 6. When happy → deploy
databricks bundle validate
databricks bundle deploy
databricks bundle run agent_langgraph
```

## Bottom line for your specific concern

> *"anything that databricks uses, make sure that is working when I deploy"*

Because the **start command, agent code, MLflow tracing, and Databricks SDK calls are all identical between local and deployed**, anything that works locally will work after deploy with **two specific exceptions to verify post-deploy**:

1. **Service principal permissions** — every Databricks resource your agent touches (model serving endpoint, Genie space, UC function, Vector Search index) must be listed in `databricks.yml` under the app's `resources:` so the SP gets access. **This is the #1 source of "works locally, fails deployed" issues.**
2. **OBO auth** (`get_user_workspace_client()`) — only works when deployed and called through the apps reverse proxy. Don't rely on it locally.

Run `uv run preflight` before every deploy and you'll catch most code-level issues before they hit Databricks.
